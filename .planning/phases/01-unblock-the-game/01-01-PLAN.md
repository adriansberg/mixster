    3. Build the URL: https://api.spotify.com/v1/playlists/${playlistId}/items?offset=0&limit=1&fields=total (first call at offset=0 with limit=1 to get total cheaply).

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → POST /api/spotify/songs/random | sessionId, selectedDefaultPlaylists, customPlaylistUris from browser — untrusted |
| server → Spotify API | Bearer token from DB; response data trusted but must be null-checked |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Tampering | sessionId in request body | accept | sessionId is a string key for DB dedup; not used for auth or authorization — no privilege escalation possible |
| T-01-02 | Spoofing | customPlaylistUris from client | accept | URIs passed to Spotify API server-side; Spotify enforces ownership — 403 returned for inaccessible playlists |
| T-01-03 | Information Disclosure | access token in raw fetch call | mitigate | getSpotifyAccessToken called server-side only; token never returned to client; existing spotifyFetch pattern maintained |
| T-01-04 | Denial of Service | retry loop with 10 attempts × 2 Spotify calls | mitigate | Early return on 403 (no exhaustion); each attempt makes at most 2 API calls; Spotify rate-limits are upstream concern (Phase 4) |
| T-01-05 | Tampering | selectedDefaultPlaylists array from client | accept | Array is filtered against server-side defaultPlaylists config — client cannot inject arbitrary URIs via this field |
| T-01-SC | Tampering | npm/pip/cargo installs | accept | Phase installs no new packages |
</threat_model>
