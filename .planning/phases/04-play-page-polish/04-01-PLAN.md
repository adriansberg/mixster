<objective>
Harden server-side Spotify integration: widen the token refresh buffer from 1 minute to 5 minutes (API-03), and add explicit 429 short-circuit guards to both raw Spotify fetches inside the songs/random retry loop so a rate-limit response surfaces as HTTP 429 with `{ rateLimited: true }` instead of silently hammering Spotify for 10 attempts (API-04, server portion).

Purpose: Mid-game token expiry forces a user to re-auth between cards; widening the buffer keeps the round flowing. The current 429 path is the worst bug class — when Spotify rate-limits us, we retry up to 10 times against the same rate-limited endpoint, escalating the throttle. Both fixes are surgical edits to existing 401/403 guard patterns.

Output:
- `src/lib/server/spotify.ts` line 29 token buffer changed from `60 * 1000` to `5 * 60 * 1000` (D-03)
- `src/routes/api/spotify/songs/random/+server.ts` gets a `status === 429` guard on both `totalResponse` and `windowResponse`, returning `json({ error, rateLimited: true }, { status: 429 })` before the generic `!ok` branch
</objective>
