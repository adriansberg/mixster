## Spacing Scale

Declared values (multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, inline badge padding |
| sm | 8px | Button gap in header bar, compact chip spacing |
| md | 16px | Card internal padding (mobile), form field gap |
| lg | 24px | Card internal padding (desktop), section spacing |
| xl | 32px | Section-to-section gap |
| 2xl | 48px | Major section breaks |
| 3xl | 64px | Page-level vertical padding (desktop) |

Exceptions:
- Touch targets for interactive buttons: minimum 44px tall
- Play/pause circular button: 64px × 64px (`w-16 h-16`) — inherited
- Inline error banner: `p-3` (12px) minimum, `p-4` (16px) preferred
- Re-auth banner: `py-3 px-4` to ensure readability at party distance

Source: Phase 3 UI-SPEC (inherited); inline banner sizing derived from existing error state `p-4 md:p-6` pattern in play page.

## New UI Elements — Phase 4

### Inline Error Banner (replaces `alert()` — UI-01)

Replaces 3 `alert()` calls in `play/+page.svelte`. Already partially implemented via `errorMessage` state and `bg-destructive/10` div, but `alert()` paths remain.

| State | Trigger | Surface | Copy pattern |
|-------|---------|---------|--------------|
| Generic error | Any API failure not covered below | `bg-destructive/10 border border-destructive` | Norwegian: problem description + action hint |
| No device | Devices fetch returns empty | Same destructive surface | "Ingen Spotify-enheter funnet. Åpne Spotify på en enhet først." |
| Playback failed | Player play/pause returns non-OK | Same destructive surface | "Avspilling feilet: [error]" |

Location: replaces the existing `{:else if errorMessage}` block — already positioned correctly below device selector in play page.

### Re-auth Inline Banner (replaces hard redirect — UI-02)

Replaces `alert() + goto('/auth/login/spotify')` pattern (3 occurrences: devices fetch, getNextSong, playSong).

| Property | Value |
|----------|-------|
| Surface | `bg-orange-500/10 border border-orange-500/50 rounded-lg p-4` |
| Heading | "Spotify-økt utløpt" (`text-base font-semibold`) |
| Body | "Logg inn på nytt for å fortsette." (`text-sm text-muted-foreground`) |
| CTA | Button: "Logg inn igjen" — `variant="outline"` — `onclick={() => goto('/auth/login/spotify')}` |
| Placement | Same position as error banner — replace `errorMessage` block |
| State flag | Add `requiresReauth: boolean` state variable to hold this independently from generic `errorMessage` |

User must click the button to initiate redirect. No automatic redirect. No `window.confirm()`.

## Interaction Contracts

### alert() Elimination (UI-01)

Three `alert()` calls in `play/+page.svelte` to eliminate:

1. `onMount` — devices fetch → `data.requiresReauth` → currently: `alert(...) + goto()`
   - Replace: set `requiresReauth = true` → render re-auth banner
2. `getNextSong` — songs/random → `error.requiresReauth` → currently: `alert(...) + goto()`
   - Replace: set `requiresReauth = true` → render re-auth banner
3. `playSong` — player/play → `error.requiresReauth` → currently: `alert(...) + goto()`
   - Replace: set `requiresReauth = true` → render re-auth banner

All three share the same state flag `requiresReauth`. Banner replaces `errorMessage` panel in the UI — mutually exclusive display.

State priority for the single display slot:
1. `requiresReauth` — re-auth banner (orange)
2. `rateLimited` — rate limit banner (yellow)
3. `showDeviceSelector && availableDevices.length > 0` — device selector panel
4. `errorMessage` — generic error banner (destructive/red)

Note: device selector appears before errorMessage so onMount device-fetch errors don't obscure the device chooser — per Plan 04-02 Task 1 (Pitfall 5 fix).

### console.log Removal (UI-03)

`console.log('data', data)` on line 62 of `play/+page.svelte` — remove. No UI change.

### No window.confirm() (Inherited from Phase 3)

Already implemented via double-press pattern on TØM HISTORIKK. No changes needed. PLAY-07 URI validation uses inline text, not confirm dialog.

### 429 Handling State Machine

```
getNextSong() called
  → fetch /api/spotify/songs/random
  → response.status === 429
    → set rateLimited = true
    → set loading = false
    → return (stop retry loop)
  → user clicks "PRØV IGJEN"
    → set rateLimited = false
    → call getNextSong()
```

No automatic retry. No exponential backoff in browser. Server may add Retry-After header — read and display wait time if present: "Spotify er overbelastet. Prøv igjen om X sekunder."
