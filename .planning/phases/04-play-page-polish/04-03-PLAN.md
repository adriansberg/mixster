<objective>
Add client-side Spotify URI/URL format validation to `src/routes/setup/+page.svelte` so an invalid paste produces an inline error message instantly — without making the round-trip to `/api/spotify/playlist/validate` (which today returns a vague server error after a real fetch). Introduces one new `$state` flag (`uriValidationError`), one new local helper (`isValidSpotifyPlaylistInput`), and one inline render slot beneath the existing playlist Input.

Purpose: Today, pasting a YouTube URL or a Spotify track URI silently fires the validate endpoint and returns a server error after a network round-trip. The server validator already exists (`parseSpotifyPlaylistId` in `src/lib/server/spotify.ts`) and remains the authoritative check — this plan adds a cheap client-side regex/URL gate that mirrors the same three accepted formats and short-circuits the form submission with an inline Norwegian message. The server still validates authoritatively on every accepted submit.

Output:
- New `$state` flag `uriValidationError = $state('')`
- New local helper `isValidSpotifyPlaylistInput(input: string): boolean` mirroring `parseSpotifyPlaylistId` logic
- `addCustomPlaylist()` entry block clears both `errorMessage` AND `uriValidationError`, then runs the validator before any `fetch` call
- Template: new `{#if uriValidationError}` block below the existing `{#if errorMessage}` block
</objective>

From src/lib/server/spotify.ts (lines 182–208, do NOT modify):
```typescript
export function parseSpotifyPlaylistId(input: string): string | null {
  if (input.startsWith('spotify:playlist:')) {
    return input.split(':')[2];
  }
  try {
    const url = new URL(input);
    if (url.hostname === 'open.spotify.com' && url.pathname.startsWith('/playlist/')) {
      const id = url.pathname.split('/')[2];
      return id?.split('?')[0] ?? null;
    }
  } catch { /* not a URL */ }
  if (/^[a-zA-Z0-9]{22}$/.test(input)) {
    return input;
  }
  return null;
}
```

Existing `addCustomPlaylist()` entry in `src/routes/setup/+page.svelte` (lines 152–187): runs `if (!playlistInput.trim()) return;`, then `errorMessage = '';`, then `addingPlaylist = true;`, then `fetch('/api/spotify/playlist/validate', ...)`. The new validation gate inserts AFTER `errorMessage = ''` and BEFORE `addingPlaylist = true`. On failure, return without firing the fetch and without flipping `addingPlaylist`.
