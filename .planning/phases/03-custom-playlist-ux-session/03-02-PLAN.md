<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser localStorage -> play page state | User-controllable; tampering only affects local UI selections (server re-validates via session + Spotify) |
| browser -> /api/spotify/history/clear | Authenticated POST, scoped to event.locals.user — server already authorizes |
| browser -> /api/spotify/songs/random | Authenticated POST with client-supplied URI list — server already authorizes by user and uses URIs only as Spotify query filters |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-02-01 | Tampering | localStorage `mixster_playlists` read in play page onMount | mitigate | `parsePlaylistState` (from Plan 01) silently resets malformed values; play page receives a valid PlaylistState shape and never crashes on garbage input |
| T-03-02-02 | Elevation of Privilege | Double-click race on TØM HISTORIKK firing duplicate /history/clear POSTs | accept | The clear is idempotent (DELETE rows by userId); duplicate POSTs are harmless. We also set `clearPending = false` before awaiting fetch, so any third rapid click goes back to the pending state rather than firing a fresh DELETE without confirmation |
| T-03-02-03 | Information Disclosure | songsPlayed counter exposing session activity | accept | Counter is client-only, scoped to one user's own session — no leakage |
| T-03-02-04 | Denial of Service | setTimeout for clearSuccess piling up across many clears | accept | Single 2-second timer per successful clear; user-triggered, bounded — no resource exhaustion |
| T-03-02-05 | Spoofing | Counter manipulation via devtools to fake songs played | accept | Counter is cosmetic only; no game logic depends on it |
| T-03-02-06 | Tampering | onblur cancellation race (user pressing the button while it is losing focus) | accept | Worst case: an extra confirmation step. Not a security issue |
| T-03-02-SC | Tampering | npm package installs | accept | No new packages installed by this plan. No legitimacy audit required. |
</threat_model>
