<objective>
Establish the consolidated `mixster_playlists` localStorage schema with Zod validation, one-time migration from the old keys, and rebuild the setup page so default and custom playlists share the same enable/disable toggle UX with immediate persistence.

Purpose: Phase 3 success depends on a single source of truth for playlist selection state. Today selections live across three keys (`mixster_selected_defaults`, `mixster_custom_playlists`, transient `selectedDefaults` state) and only persist at game start. This plan consolidates them, makes toggles persistent on every interaction, and brings custom playlists to feature parity with defaults.

- New `src/lib/config/playlist-state.ts` module exporting schema + helpers (D-01, D-03, D-04)
- Refactored `src/routes/setup/+page.svelte` reading/writing the new schema (D-05, D-07, D-08, D-09) and migrating old keys on first load (D-04)
</objective>

New schema this plan introduces (in src/lib/config/playlist-state.ts):
```typescript
export const STORAGE_KEY = 'mixster_playlists';
export const PlaylistStateSchema = z.object({
  version: z.literal(1),
  defaultSelected: z.array(z.string()),
  custom: z.array(z.object({
    id: z.string(),
    name: z.string(),
    uri: z.string(),
    trackCount: z.number(),
    enabled: z.boolean()
  }))
});
export type PlaylistState = z.infer<typeof PlaylistStateSchema>;
export const DEFAULT_PLAYLIST_STATE: PlaylistState;
export function parsePlaylistState(raw: string): PlaylistState;  // silent fallback
export function migrateOldKeys(): void;  // one-time, browser-only
```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser localStorage -> app state | User-controllable persistent storage; can be edited via devtools or restored from another browser profile |
| browser -> /api/spotify/playlist/validate | Untrusted input (playlist URL/URI) crosses to server — already validated server-side, no change in this plan |
| browser -> /api/spotify/songs/random | Untrusted client-supplied playlist URI list; the server already authorizes by `event.locals.user` and uses URIs only as filter inputs to Spotify (no SSRF surface) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01-01 | Tampering | localStorage `mixster_playlists` key | mitigate | Zod `safeParse` in `parsePlaylistState` discards malformed payloads silently and falls back to DEFAULT_PLAYLIST_STATE — no app crash, no privilege gained because the key only drives client UI selections that are re-validated server-side by `/api/spotify/songs/random` (whitelists default URIs by id) |
| T-03-01-02 | Tampering | Old migrated keys (`mixster_custom_playlists`, `mixster_selected_defaults`) | mitigate | JSON.parse wrapped in try/catch inside `migrateOldKeys`; malformed values fall through to defaults and old keys are unconditionally removed at the end so a corrupt blob cannot persist across sessions |
| T-03-01-03 | Information Disclosure | Custom playlist URIs stored client-side | accept | URIs are user-supplied public Spotify identifiers, already present in `customPlaylists` localStorage today; no PII added by this plan |
| T-03-01-04 | Spoofing | User pasting another user's private playlist URL | accept | Spotify's `/playlists/{id}` endpoint enforces server-side access control via the user's OAuth token — out of scope to defend further here (covered by v2 RES-02) |
| T-03-01-05 | Denial of Service | Extremely large pasted JSON in old keys triggering parse-time blowup | mitigate | `JSON.parse` is bounded by localStorage size limits (~5–10MB per origin); malformed input falls through to defaults and removes the offending key — single transient failure max, not a loop |
| T-03-01-06 | Repudiation | localStorage write race between tabs | accept | Solo-user / single-screen party-jukebox context (PROJECT.md). Last-write-wins is acceptable; no audit trail needed |
| T-03-01-SC | Tampering | npm package installs | accept | No new packages installed by this plan (zod already in package.json ^4.1.12, nanoid already present). No package legitimacy audit required. |
</threat_model>
